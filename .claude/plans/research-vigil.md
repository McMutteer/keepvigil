# Vigil — Research Findings

**Date:** 2026-03-01
**Researched by:** Master Plan skill

## Market & Problem Validation

### The Problem — Quantified

Analysis of **35 real PRs** across 8 repositories (binance-trading-platform, nqual5, stripe-gateway, console-miia, whatsmiia, miia-console, n8n_dashboard, env-vault) reveals:

- **94% of PRs** generated by AI agents include a "Test Plan" section
- **70% of test plan items** are left unchecked `[ ]` — never executed
- **30% pre-checked** `[x]` are items the AI ran locally (lint, build, unit tests)
- **100% use markdown checkbox format** (`- [ ]` / `- [x]`)
- Format always titled `## Test plan` or `## Test Plan`

### Test Plan Item Categories

| Category | % of items | Automatable? | Example |
|----------|-----------|-------------|---------|
| CI/Build checks | ~25% | Already automated in CI | `npm run build` passes |
| UI functional flows | ~30% | Yes (Playwright E2E) | "Click 'Conectar' → QR code appears" |
| API integration | ~15% | Yes (HTTP assertions) | "POST /api/agents → verify agent created" |
| Visual/UI verification | ~15% | Partially (visual regression) | "Skeleton appears briefly then fades" |
| SEO/Metadata | ~5% | Yes (HTML parsing) | "JSON-LD valid for Google Rich Results" |
| Infrastructure/DevOps | ~10% | Partially | "docker build produces image < 50MB" |

### Key Patterns in Test Plans

1. **Action-result pairs:** Nearly all items follow "Do X → verify Y" or "Do X -- should Y"
2. **Explicit "Manual:" prefix:** Used to separate from CI checks
3. **Exact strings to search for:** Include UI text in Spanish/English
4. **Graceful degradation checks:** "Unknown coins get empty AssetMeta (no crashes)"
5. **Security-specific items:** IDOR checks, auth bypass, ownership verification
6. **No performance benchmarks:** Never include measurable perf criteria

### Specificity Spectrum

| Level | Example | Frequency |
|-------|---------|-----------|
| High | "Send 6 requests → 6th should return 429" | ~40% |
| Medium | "Verify skeleton grid appears briefly then fades" | ~45% |
| Low | "Verify responsive en mobile y desktop" | ~15% |

### GitHub Actions for PR Checklists

Several exist but NONE execute tests:

| Action | What it does | Gap |
|--------|-------------|-----|
| check-pull-request-description-checklist | Validates checkbox state | Only checks if boxes are ticked |
| Require Checklist | Fails PR if items incomplete | Enforcement only, no execution |
| gh-checklist | Exports checklist as variables | Raw data, no test execution |
| check-list-extract | Exports results as Action variables | Generic parsing |

**Key finding:** Building blocks for parsing exist. What does NOT exist is anything that feeds parsed items into a test execution engine.

### User Sentiment

The "AI Paradox" in software development:
- AI helps write 40% more code (GitHub data)
- 82% of teams deploy weekly (GitLab 2025)
- But testing hasn't scaled proportionally
- Test plans are "honor system" — decoration, not verification

### Market Assessment

- **AI-enabled testing market:** $1.01B in 2025, projected $4.64B by 2034 (18.3% CAGR)
- **Broader automation testing market:** significantly larger
- **Key driver:** Rising demand for no-code/codeless testing and CI/CD integration
- **Window:** Real but narrow — well-funded incumbents could move into this space

## Technical Landscape

### Core Technical Components

1. **PR Parser:** Extract test plan from PR description markdown
2. **NL Interpreter:** Translate natural language items to executable test specifications
3. **Test Executor:** Run tests against a preview deployment or local build
4. **Reporter:** Post results back to the PR as checks/comments

### Proven NL → Test Execution Approaches

| Approach | Used By | How |
|----------|---------|-----|
| LLM + Playwright | Shortest | Claude API interprets NL, generates Playwright actions, executes |
| Custom NLP engine | testRigor | Proprietary NLP parses structured English into test commands |
| LLM + Cypress | Cypress cy.prompt() | AI translates NL to Cypress commands at runtime |
| LLM + Browser Agent | Momentic | AI agent navigates browser based on NL description |

**Recommendation:** Use Claude API as interpreter + Playwright as executor (like Shortest). The innovation is in the **orchestration**, not the execution engine.

### Test Category Execution Strategies

| Category | Execution Strategy | Confidence |
|----------|-------------------|------------|
| Build/lint/test commands | Direct shell execution | HIGH |
| UI functional flows | Playwright + Claude vision for verification | HIGH |
| API assertions | HTTP client + assertion library | HIGH |
| Visual regression | Playwright screenshots + pixel/AI comparison | MEDIUM |
| SEO/Metadata | HTML parsing + schema validation | HIGH |
| Responsive/viewport | Playwright viewport emulation | HIGH |
| Touch/mobile | Playwright touch emulation | MEDIUM |
| Infrastructure | Docker/shell commands in sandbox | MEDIUM |

### Key Libraries & APIs

| Library | Purpose | Why |
|---------|---------|-----|
| Playwright | Browser automation | Most complete, multi-browser, built-in assertions |
| Anthropic SDK | NL interpretation | Claude for understanding test intent and visual verification |
| Octokit | GitHub API | PR parsing, check runs, comments |
| Probot/GitHub App framework | GitHub App | Webhook handling, installation management |
| cheerio | HTML parsing | SEO/metadata validation |
| ajv | JSON Schema validation | JSON-LD validation |

### Infrastructure Considerations

| Concern | Approach |
|---------|----------|
| Browser execution | Containerized Playwright (Docker) |
| Preview deployments | Integrate with Vercel/Netlify preview URLs |
| Cost per execution | LLM calls (Claude) + browser time |
| Concurrency | Queue-based execution per PR |
| Security | Sandboxed browser, no access to prod |

### Pitfalls & Risks

| Pitfall | Why it's dangerous | Mitigation |
|---------|-------------------|------------|
| Ambiguous test items | "Verify responsive" — what constitutes pass? | Classify items by confidence, flag ambiguous ones |
| LLM hallucination | AI might "pass" a test it didn't actually verify | Visual evidence (screenshots), deterministic checks where possible |
| Flaky browser tests | E2E tests are notoriously flaky | Retry logic, smart waits, screenshot-on-fail |
| Cost explosion | Claude API + browser per PR could be expensive | Cache, batch, tier by plan |
| Preview deployment dependency | Need a running app to test against | Support local builds via CLI, not just preview URLs |
| Rate limiting | GitHub API limits for heavy users | Efficient API usage, caching |

## Best Practices

### Domain-Specific Patterns

1. **GitHub Check Runs** are the standard for CI results display
2. **PR comments** with collapsible sections for detailed results
3. **Status badges** for quick pass/fail visibility
4. **Annotation API** for inline file-level feedback

### Data Model Patterns

- Test plan items → structured test specifications
- Execution results → screenshots, logs, assertions
- Historical data → flakiness tracking, coverage trends

### Compliance & Security

- No access to production environments
- Browser sandbox — isolated execution
- API tokens encrypted at rest
- GDPR: minimal data retention (execution logs)
- SOC 2 considerations for enterprise tier
